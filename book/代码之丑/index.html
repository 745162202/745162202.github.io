长函数与方法内联

　　长函数把多个业务处理流程放在一个函数里实现，把不同层面的细节放到一个函数里实现，让阅读这端代码的人望而却步。

　　长函数的原因主要使两个：

　　第一种把代码写成长函数原因是：长函数比较平铺直叙非常的直白把自己思考的一点点罗列出来，没有关注代码的分离。

　　另一种是因为觉得单个长函数性能好，把函数写长是为了获更好的性能。 是的每次调用方法可能涉及到压栈、寄存器中的值替换、指针需要指向被调用代码的第一行的地址再执行等操作浪费性能。确实这会导致一部分性能损失，但是代码的可维护性比性能优化要优先考虑（代码还是需要满足共识的编码规范），剩下就需要当前程序性能不足以满足需要时我们再来做相应的测量，找到焦点进行特定的优化。况且随着编译器越来越好，优化也越来越多，现代 JVM 都支持方法内联，把这些热点方法再 JVM 层面内联起来，使性能和长函数相似。

长参数列表

　　很多人编写代码还会是大量长参数传递的模式，这会导致两个比较难搞的问题：

  1. 不够聚合，比如某个参数需要增加前缀或者后缀，你需要去每个相关的地方修改
  2. 业务逻辑修改增加了参数，可能导致多个地方的修改把新增的参数一步一步传递下去

　　如果可以通过封装把参数封装成对象，那么对业务模型的变化修改起来很方便并且更加聚合加前缀等问题可以再模型中 get 方法中统一处理。

可变对象和可变数据

　　可变的对象让它们变得不可控，对象中到处都是 set 方法再对象传递的过程中可以被任意修改，没有人可以确认当你使用的时候它是什么值。推荐默认不写 set 只有业务真的需要改变在加入对于的操作的方法。

　　可变数据这个做到的应该很少数据通过 final 修饰它比可变对象好点但还是坏味道。

依赖混乱

　　这章中的防腐层可以仔细品下虽然很多人都会犯这类问题。主要讲的就是关于 Controller DTO 是否可以在 Service 直接使用问题。总结就是不推荐，请求对象传导到业务代码中，造成了业务与外部接口的耦合，也就是业务依赖了一个外部通信协议。引入一个防腐层，将业务和外部通信协议隔离开来。
