未命名

发表于 2023-02-09

0.背景

​ k8s 下运行 ElastcSearch， 通过 ElastcSearch API 获取 ElastcSearch进程 CPU 使用率，它获取的确是宿主机下的该容器中 ElastcSearch 进程 CPU 使用率导致监控不准。

下图宿主机 CPU 为4核，ElastcSearch 镜像进程 CPU 100%，但是 getProcessCpuLoad 获取确实 25%左右。

![img](D:\Markdown\K8S\assets\记录踩坑：OpenJDK 获取进程CPU使用率不准问题.assets\1655535154117-32d0cea7-a639-4123-b527-70aefaa4a6ac.jpeg)

1. 问题排查

​ 通过查看 ElastcSearch 源码发现 ElastcSearch 是通过 JDK 的 OperatingSystemMXBean 的 getProcessCpuLoad 方法获取进程 CPU使用率， getProcessCpuLoad 最后会调用本地方法。

通过 OpenJDKd 的 issues 中找到了这个 bug 的 fixed 信息是 2021 年 7 月解的这个问题 getProcessCpuLoad reports incorrect process cpu usage in containers 在上可以看到修复了 8、11、17 这当前长期维护的版本。但是 OpenJDK 官网中并没有提供修复的 jdk11最新的版本下载，如果需要解决该问题只能升级 OpenJDK 到 17.0.2 或者使用微软或者阿里维护的 OpenJDK。

​ 重写构建镜像测试一下获取的进程 CPU 使用率准确了，记录下方便帮助遇到同样问题的人。

Google软件测试之道心得

发表于 2022-05-08 分类于 读书笔记

角色定义

软件开发工程师 SWE(Software Engineer)

​ 软件开发工程师是一个传统上的开发角色，创建设计文档，编写与测试代码包括测试驱动的设计、单元测试、参与构建各种大小规模的测试等。

软件测试开发工程师 SET(Software Engineer in Test)

​ 我觉的可以把他理解为产品专家，是一个编码能力很强的程序员，可以写功能代码，同时也是一个能力很强的测试者，可以测试任何产品，辗转于各个项目组。与软件开发工程师深度合作实际工作内容也会有所重叠，它是在软件中后期进场，进场后会查看所以迭代的设计文档了解当前系统设计，后续审核设计文档并且推进编写测试程序和框架，SET更加关注于质量的提升和测试覆盖率的增加。

测试工程师 TE(Test Engineer)

​ 测试工程师也称作“用户开发者”，因为 TE 需要站在用户角度了解使用产品，负责端到端测试用例编写，对每一个重要的缺陷都要增加一个测试用例与之对应，并且跟踪推动缺陷修复。测试工程师也需要查看历史的功能设计文档，并且对后续的测试用例模块进行评审。

​ 测试工程师重要品质测试人员不该对测试文档过于珍爱，糟糕的测试用例需要被抛弃，而最后留下来的是更好的测试用例。测试重要的一面是做确认，完成测试计划并且确认，使程序崩溃并不总是测试工程师的目标。以极端的输入数据来测试软件并使之出错这很意义，但更有意义的是用不那么极端的输入，一遍又一遍地测试用以模拟真实的使用场景，确保这些通用条件下软件的运行不会出错。在面试时候Google也会特意寻找这种有正面的测试观的测试人员。

​ 测试工程师还需要对软件风险进行把控，需要通过对失败频率和影响范围对风险进行分析打分，对暴露出来的分险进行用户故事和测试用例的覆盖，如果已经覆盖需要考虑当前测试用例是否足够，是否需要增加额外的测试。

ES Lucene marge和 forceMerge 详解

发表于 2022-05-08 分类于 elasticsearch

Lucene的机制当内存中有数据的情况下，每 1s 生成一个segment。当segment生成后这个数据就可以被检索到了,segment增多会导致占用文件句柄增加，消耗大量资源，Lucene通过把segment合并来解决这个问题。Lucene通常有三种类型的合并：自动的合并naturalMerge(下文称为 Merge)、手动的forceMerge和forceMergeDeletes。merge是ES为了提升查询性能、回收已经被删除的doc，在后台定期调度的merge操作。对索引的任何增删改操作都有可能触发merge。而forceMerge手动触发的merge操作，目的是减少 segment 数量和回收已经被删除的 doc，提升查询性能，forceMerge比较暴力最好是在索引为只读状态后使用。forceMergeDeletes合并存在删除的内容占10%以上的segment。

k8s为什么需要设计一个Pod

发表于 2022-05-08 分类于 k8s

Pod 是 Kubernetes 项目的原子调度单位，pod 中容器直接可以共享存储和网络，可以很方便的进行文件交互。

0. 为什么需要 pod

​ 我们可以把 Kubernetes 比作操作系统，容器镜像就是这个系统里的安装包，那么容器就可以类比为进程。在一个真正的操作系统里，进程并不是孤苦伶仃地独自运行的，而是以进程组的方式，有原则地组织在一起。

​ 一个主进程下面有多个线程一起协作下有条不紊的工作，Linux 操作系统语境下的线程可以称之为轻量级进程，他们组成了一个线程组。当我们把容器类比为进程，那么 Kubernetes 项目其实就是将进程组的概念映射到了容器技术中称之为 pod。
