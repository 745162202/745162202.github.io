k8s为什么需要设计一个Pod

发表于 2022-06-18 分类于 k8s

Pod 是 Kubernetes 项目的原子调度单位，pod 中容器直接可以共享存储和网络，可以很方便的进行文件交互。

0. 为什么需要 pod

​ 我们可以把 Kubernetes 比作操作系统，容器镜像就是这个系统里的安装包，那么容器就可以类比为进程。在一个真正的操作系统里，进程并不是孤苦伶仃地独自运行的，而是以进程组的方式，有原则地组织在一起。

​ 一个主进程下面有多个线程一起协作下有条不紊的工作，Linux 操作系统语境下的线程可以称之为轻量级进程，他们组成了一个线程组。当我们把容器类比为进程，那么 Kubernetes 项目其实就是将进程组的概念映射到了容器技术中称之为 pod。

ZooKeeper一致性算法——ZAB协议

发表于 2021-12-03 分类于 zookeeper

0. ZAB协议基础介绍

​ ZAB（ZooKeeper 原子消息广播协议）一种类 Paxos 算法，ZAB 协议是为分布式协调服务 ZooKeeper 专门设计的一种支持崩溃恢复的原子广播协议。

​ ZAB 协议的核心是定义了对于那些会改变 ZooKeeper 服务器数据状态的事务请求的处理方式，即：所有事务请求必须由一个全局唯一的服务器来协调处理，这样的服务器被称为 Leader 服务器，而余下的其他服务器则成为 Follower 服务器或是 Observer 服务器（与 Follower 的区别就是不参加选举）。Leader 服务器负责将一个客户端事务请求转换成一个事务 Proposal（提议），并将该 Proposal 分发给集群中所有的其他服务器。之后 Leader 服务器需要等待超过半数的服务器进行了正确的反馈后，那么 Leader 就会再次将这些服务器发送 Commit 消息，要求其将前一个 Proposal 进行提交。

CAP 定理和当前软件的 CPA 的实现

发表于 2021-05-21 分类于 分布式系统

0. CAP 定理

随着计算机系统规模变得越来越大，将所有业务集中在一个或多个小型机的集中式架构下，已经不能满足当前大型互联网公司的快速发展，而且集中式架构也存在诸多单点问题。为解决这一系列问题分布式架构脱颖而出，采用廉价的 PC机 通过网络通信完成数据的计算和存储，而且能保证 7 * 24 小时不间断对外提供服务。

分布式系统解决了很多单机场景下无法解决的问题，但是也引入了新的挑战，比如无法满足单机下被实践证明非常成熟的 ACID（原子性，一致性，隔离性，持久性） 模型来保证数据强一致性。在分布式下可用性和一致性之间永远无法存在一个两全其美的方案，于是在分布式架构下构建兼顾可用性和一致性的系统成为一个难题。随着实践慢慢孕育出列分布式系统经典理论 CAP 定理。
