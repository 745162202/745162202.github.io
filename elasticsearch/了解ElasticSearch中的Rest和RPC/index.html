了解 ElasticSearch 中的 Rest 和 RPC

发表于 2023-11-05 分类于 elasticsearch 阅读次数：

ElasticSearch源码版本 7.10.0

本文章主要通过一个创建索引的视角浏览 ElasticSearch (ES) 从请求接受到 RestAction 再到 TransportAction 流程。大家都知道 ES 节点启动后他会开启两个端口分别是监听接受客户端请求的 9200、处理节点内部请求的 9300。那么第一个问题浮现在我们眼前一个请求到了 ES 将如何处理请求呢?

答案就是 Action，ES 中有两个分别是处理客户端请求的 RestAction 和处理节点内部请求 TransportAction 统称 Action。其实 ES 在 Action 中存规则，了解规则我们就快速找到自己找到自己看的源码，下面我们通过创建索引流程来介绍下，想直接看结论的直接看总结。

RestAction

链路：

关键链路

Netty4HttpRequestHandler 接受请求完成后我们需要特别关注的是 RestController 见名识意他是处理 Rest 请求的， RestController 实际做的就是 HTTP 请求中 Header 校验和最重要的部分通过 URL 获取所有可能的 MethodHandlers, 再通过 Request 的 Method 找到对应的 Handler (具体 RestAction)，最终执行这个 Action 的动作。

上面介了通过 URL 和 Method 在 Handlers 找到对应的 Handler，那 Handlers 的映射数据是什候把注册进去的呢? 答案就是 Node 初始化，Node 的 ActionmModule 初始化中顺便构建一个 RestControler， Node 会调用ActionModule#initRestHandlers 完成 RestAction 和 ActionPlugin 的注册。

RestControler#registerHandler 最终注册的地方，我们 RestCreatelndexAction 举例 restHander.routes() 获取的是 singletonList(new Route(PUT,”{/index}”)，当客户端发送PUt /index 请求过来那么我们最终会在 RestCreatelndexAction#orepareReuest 被处理，RestiCreatelndexAction#prepareRequest 就是第-个规则，详细见下面三段码。这里可以看出 RestActon 采用的典型的命令模式解藏调用者和执行者，不同的 RestAction 在自己不同的代码中完成。新增接口增加一个 RestAton 并且把类写到 intBestHandlears 中。

TransportAction

链路

介绍

上面介绍的 RestAction 里面的 RestCreatelndexAction#PrepareReouest 最后一行 client.admin().indicesl().create 中 create 方法调用 IndicesAdmin 类的 create 方法，再调用 execute 方法的入参 CreatelndexAction.INSTANCE。

TransportAction 这块能路没有什么关键代码，比较关键的就是 TransportMasteNodeAction，因为例子是创建索引所以我们到了 TransportiMlasterNodeAction 它是所有Master 处理请求的基类（如不需要 Master 操作关键请求在 doExecute 中），这里主要逻辑就是判断当前节点是不是主 Master 节点如果不是的话需要转发到主 Master 节点。这一步是放到 same 线操作执行 MasterOperation。

上面没讲一个非常关键的代码 Transportction 和 ActionType 如何做关的 TransportAction(actin) 可以直接获取对应的 TransportAction？

TransportAction 关联也是在 ActionModule 时完和 ActionType 的映射，并借助了 Guice (Guice 是 Google 的赖注入框架) 完成实例创建，ES 直接移植 Guice 源码到代码库中维护，这样可以更好地适配

自身的架构和需求，并且能够在需要时自由修改和定制代码。

TransportAction 是在 ActionModule#setupAction 中初始化注册整理出所有 ActionType 和 ActionHandler (TransportAction) 的映射，通过 actions.register(MainAction.NSTANCE, TransportMainAction.class)，这里 TransportAction 实例并有没真正创建出来。

ES 中通过 ActionModule#configure 最终完成 ActionType 和 TransportAction 实例之间的映射，Guice 中 key 为 Map<ActionType, TransportAction>，详细了解的可以参考文档Guice 官网介绍 MapBinedr 规则

Guice 注册完成通过调用 NodeCient 的 initialize把 uice 中 key 为 Map<ActionType,TransportAction> 的获取出来放入 NodeClient actions 中，这里集合第一步 RestAction 中的 RestCreatelndexAction#prepareRequest 对应起来了。

MetaDataCreateIndexService

TransportCreatelndexAction#masterOperation 最后会调用到 MetadataCreatelndexService#createlndex 这里只是提交一个任务到一个单线程的线程池里面 applyCreatelndexRequest 真正执行创建索引的流程并返回新的 ClusterState，在任务接受到新的 ClusterState 就会去比较新老 ClusterState，ClusterState不一样会重新生成 Route 和 MateData 并且版本号会加 1，最后通过增量或者全量的方式发送给别的节点。

总结：

  1. Rest***Action 直接看 prepareRequest
  2. Transport***Action 不需要 Master 处理的请求直接看 doException, 需要 Master 操作的直接看 masterOpration
